## Overview

The schema is organized around these core entities:

```
User → creates → Projects
Projects → contain → Tasks
Users → request or get assigned → Tasks
Tasks → have → Comments
Users → generate → Invites → bring in → new Users
```

Auth-related models (`Session`, `Account`, `Verification`) are managed by [Better Auth](https://www.better-auth.com) and should not be modified manually.

## Enums

### `Role`
Controls platform-level permissions for a user.

| Value | Description |
|-------|-------------|
| `User` | Default role. Standard community member. |
| `SuperAdmin` | Full platform access. Can manage all users and projects. |

### `UserStatus`
Controls whether a user can access the platform.

| Value | Description |
|-------|-------------|
| `PENDING` | Default status. User has signed up but not yet approved by a SuperAdmin. |
| `ACTIVE` | Approved. Full access to the platform. |
| `BANNED` | Revoked access. Cannot log in or interact with the platform. |

### `TaskStatus`
Represents where a task sits on the Kanban board.

| Value | Description |
|-------|-------------|
| `TODO` | Task created, not started. |
| `IN_PROGRESS` | Someone is actively working on it. |
| `IN_REVIEW` | Work done, waiting for maintainer review. |
| `COMPLETED` | Reviewed and closed as done. |

### `TaskRequestStatus`
Tracks the state of a user's self-assign request.

| Value | Description |
|-------|-------------|
| `PENDING` | Request submitted, awaiting maintainer decision. |
| `APPROVED` | Maintainer approved. User is now assigned to the task. |
| `REJECTED` | Maintainer rejected the request. |

## Models

### User

Represents a member of the Builders Collective community. Auth fields are managed by Better Auth. Custom fields extend the base user.

| Field | Type | Notes |
|-------|------|-------|
| `id` | `String` | Primary key. Set by Better Auth. |
| `name` | `String` | Display name. |
| `username` | `String?` | Unique handle (e.g. `@john`). Optional at signup, required before using the platform. |
| `displayUsername` | `String?` | Visual version of username. Managed by Better Auth username plugin. |
| `email` | `String` | Unique. Used for login. |
| `emailVerified` | `Boolean` | Set by Better Auth after verification. |
| `image` | `String?` | Avatar URL. |
| `role` | `Role` | Platform-level role. Defaults to `User`. |
| `status` | `UserStatus` | Account status. Defaults to `PENDING` until approved by a SuperAdmin. |
| `deletedAt` | `DateTime?` | Soft delete timestamp. If set, user is considered deleted but data is retained for stats. |
| `createdAt` | `DateTime` | Auto-set on creation. |
| `updatedAt` | `DateTime` | Auto-updated on every change. |

**Relations:**

| Relation | Description |
|----------|-------------|
| `sessions` | Auth sessions. Managed by Better Auth. |
| `accounts` | OAuth accounts. Managed by Better Auth. |
| `maintainedProjects` | Projects this user created and maintains. |
| `assignedTasks` | Tasks currently assigned to this user. |
| `createdTasks` | Tasks this user originally created. |
| `taskRequests` | Self-assign requests made by this user. |
| `comments` | Comments written by this user. |
| `invites` | Invite links generated by this user. |
| `usedInvite` | Invite links this user has redeemed. |

### Project

A project created by a community member. Has a Kanban board of tasks.

| Field | Type | Notes |
|-------|------|-------|
| `id` | `String` | UUID primary key. |
| `slug` | `String` | Unique URL-friendly identifier. Generated from title in application logic. |
| `title` | `String` | Project name. |
| `shortDescription` | `String` | One-line summary. Used on project cards and listings. |
| `fullDescription` | `String` | Full project description. Supports markdown. |

| `maintainerId` | `String` | FK to User. The creator and owner of this project. |
| `deletedAt` | `DateTime?` | Soft delete timestamp. Deleted projects are hidden but retained for stats. |
| `createdAt` | `DateTime` | Auto-set on creation. |
| `updatedAt` | `DateTime` | Auto-updated on every change. |

**Relations:**

| Relation | Description |
|----------|-------------|
| `maintainer` | The User who owns this project. |
| `tasks` | All tasks belonging to this project. |
| `taskRequests` | All self-assign requests made within this project. |

**Indexes:** `maintainerId`, `deletedAt`

### Task

A unit of work within a project. Lives on the Kanban board.

| Field | Type | Notes |
|-------|------|-------|
| `id` | `String` | UUID primary key. |
| `title` | `String` | Short task title. |
| `description` | `String` | Full task description. |
| `status` | `TaskStatus` | Kanban column. Defaults to `TODO`. |
| `projectId` | `String` | FK to Project. Cascades on project delete. |
| `assigneeId` | `String?` | FK to User. Nullable — tasks start unassigned. |
| `createdById` | `String` | FK to User. Who originally created this task. |
| `deletedAt` | `DateTime?` | Soft delete timestamp. |
| `createdAt` | `DateTime` | Auto-set on creation. |
| `updatedAt` | `DateTime` | Auto-updated on every change. |

**Relations:**

| Relation | Description |
|----------|-------------|
| `project` | The Project this task belongs to. |
| `assignee` | The User currently assigned to this task. Optional. |
| `createdBy` | The User who created this task. |
| `comments` | All comments on this task. |

**Indexes:** `projectId`, `assigneeId`, `createdById`, `status`, `(projectId, status)`

> The composite index on `(projectId, status)` optimises the most common query — fetching all tasks in a project filtered by Kanban column.

### TaskRequest

A request made by a user to self-assign to a task. Must be approved or rejected by the project maintainer.

| Field | Type | Notes |
|-------|------|-------|
| `id` | `String` | UUID primary key. |
| `title` | `String` | Title of the task being requested. |
| `description` | `String` | What the user plans to do. |
| `status` | `TaskRequestStatus` | Defaults to `PENDING`. |
| `projectId` | `String` | FK to Project. Cascades on project delete. |
| `createdById` | `String` | FK to User. Who made the request. |
| `createdAt` | `DateTime` | Auto-set on creation. |
| `updatedAt` | `DateTime` | Auto-updated on every change. |

**Indexes:** `projectId`, `createdById`, `status`

> TaskRequests are hard deleted when withdrawn by the user. No soft delete needed here as they carry no stats value once withdrawn.

### Invite

An invite link generated by a community member to bring someone new into Zync. New signups require a valid invite code.

| Field | Type | Notes |
|-------|------|-------|
| `id` | `String` | UUID primary key. |
| `code` | `String` | Unique invite code. Included in the invite URL. |
| `createdById` | `String` | FK to User. Who generated this invite. |
| `usedById` | `String?` | FK to User. Who redeemed this invite. Nullable until used. |
| `maxUses` | `Int` | How many times this code can be used. Default `1` for regular users. SuperAdmins can set higher. |
| `useCount` | `Int` | How many times this code has been used so far. Defaults to `0`. |
| `expiresAt` | `DateTime` | Auto-set to 2 hours after creation by default. |
| `createdAt` | `DateTime` | Auto-set on creation. |

**Relations:**

| Relation | Description |
|----------|-------------|
| `createdBy` | The User who generated the invite. |
| `usedBy` | The User(s) who redeemed the invite. |

**Indexes:** `code`, `createdById`

**Invite flow:**
1. A member generates an invite — a unique code is created and a shareable link is sent to the invitee
2. Invitee signs up using the code
3. If invited by a regular `User` → new account status is `PENDING`, awaiting SuperAdmin approval
4. If invited by a `SuperAdmin` → new account status is `ACTIVE` immediately
5. `useCount` is incremented on each successful signup
6. Code becomes invalid when `useCount >= maxUses` or `expiresAt` has passed

## Design Decisions

**Soft deletes over hard deletes**
Users, Projects, Tasks, and Comments use a `deletedAt` timestamp instead of being removed from the database. This preserves platform stats (e.g. total projects created, tasks completed) even after deletion. Hard deletes are reserved for SuperAdmin actions or cascade cleanup.

**Cascade strategy**
Database-level `onDelete: Cascade` is used only when a parent record is hard deleted — for example, deleting a User hard deletes their Sessions and Accounts. For soft deletes, cascade logic is handled in application code to ensure `deletedAt` is propagated correctly to child records.

**UUIDs over auto-increment**
All custom models use `uuid()` as the primary key. This produces cleaner URLs, is safer to expose publicly, and is consistent across all models.

**TaskRequest is separate from Task**
A `TaskRequest` is a proposal — it becomes a `Task` only after maintainer approval. Keeping them separate preserves the full request history and makes the approval workflow explicit.

**Role on User, not a separate table**
Since Zync only has two platform-level roles (`User` and `SuperAdmin`), a simple enum on the User model is sufficient. A separate roles table would be unnecessary complexity.

**Tech stack as a string array**
`techStack String[]` on Project uses Postgres's native array type. Prisma supports filtering with `has` operator, making stack-based filtering straightforward without needing a separate join table.
